# rn-expo-boilerplate Agent Guide

## Expo SDK v56

**Read the exact versioned docs** at https://docs.expo.dev/versions/v56.0.0/ before writing any code. The SDK has changed significantly across versions — do not assume prior Expo knowledge applies.

Notable experiments enabled: `typedRoutes`, `reactCompiler`.

## Setup before dev

```bash
pnpm install
pnpm run project:setup   # interactive: pick API provider + toggle features
pnpm start               # Expo dev server
```

`project:setup` is mandatory — it generates contract types, feature flags, and provider wiring. Run it again to reconfigure.

## DO NOT edit generated files

Files with `// @generated by pnpm setup - DO NOT EDIT` are written by `project:setup` and will be overwritten. These live in:

- `src/lib/setup/generated/*`
- `src/lib/api/providers/generated/*`
- `src/app/generated/*`
- `src/features/generated/*`

The setup wizard also writes `*.setup.bak` backups if it finds manual edits in managed paths.

## Verification commands (run in CI order)

```bash
pnpm lint       # expo lint (ESLint flat config)
pnpm typecheck  # tsc --noEmit (TypeScript ~6.0)
pnpm test       # jest (babel-jest, node env, @/ path mapping)
```

CI runs them in that order. Run locally before committing.

Other useful scripts:

- `pnpm format` — prettier write
- `pnpm test:watch` — jest in watch mode
- `pnpm run project:health` — diagnostic (flags: `--strict`, `--warn`, `--json`, `--output-file`)
- `pnpm run semantic-release` — manual release trigger

## Architecture notes

- **Path alias**: `@/` → `./src/*`, `@/assets/*` → `./assets/*` (configured in tsconfig.json and jest.config.js)
- **Single-package**: `pnpm-workspace.yaml` exists but has no `packages` array — it only configures `allowBuilds`
- **Router entry**: `"main": "expo-router/entry"` in package.json
- **Route groups**: `src/app/(public)/` (sign-in) and `src/app/(protected)/` (requires hydrated auth session)
- **Auth guard**: protected layout waits for store hydration, then checks `accessToken`; redirects to `/sign-in` if absent
- **Auth store**: Zustand + SecureStore persistence; pattern is `hydrateSession()` → check `isHydrated` → check token
- **API client**: Axios with circuit breaker, 401 retry with token refresh coalescing, exponential backoff on retryable statuses (408, 425, 429, 500, 502, 503, 504)
- **react-query**: staleTime 30s, retry 1, refetchOnWindowFocus false
- **Observability**: provider contracts with noop fallback (safe degradation); Sentry + PostHog wired when env keys present
- **i18n**: i18next + react-i18next, EN/FR resources, no hardcoded UI copy in core screens
- **Theme**: custom tokens in `src/theme/` (primitives + semantic layers), global CSS vars for web

## Commit rules

- **Conventional commits only** (`feat:`, `fix:`, `chore:`, etc.) — enforced by commitlint + lefthook pre-commit hook
- Commits on `main` publish stable releases, on `develop` publish beta prereleases (semantic-release)
- Do NOT push `--force` to `main` or `develop`
- Pre-commit hooks (lefthook): lint-staged → eslint --fix + prettier, gitleaks (skips locally if not installed), dep audit (non-blocking)

## Environment

- `EXPO_PUBLIC_API_URL` — base URL for the shared API client
- `EXPO_PUBLIC_FLAGS_ENDPOINT` — feature flags endpoint path
- `.env*.local` is gitignored; `.env` is tracked (for template values)
- `expo-env.d.ts` is gitignored (auto-generated by Expo on dev start)
- Env contract is validated at boot via `validateRuntimeSetup()`; missing keys cause observability to fall back to no-op

## Build & release

- **EAS profiles**: `preview` (internal distribution), `production` (auto-increment)
- **Web output**: static (`"web.output": "static"` in app.json)
- **iOS scheme**: `rnexpoboilerplate`
- **semantic-release** on push to `main` and `develop` branches (`.releaserc.json`)
- **Maestro E2E flows** in `.maestro/`; CI validates via `maestro test --dry-run`

## Testing quirks

- Jest config: `testEnvironment: "node"`, `babel-jest` transform, transforms `expo-localization` from node_modules
- Jest `@/` mapping replicates tsconfig paths
- Integration tests may require env vars from `.env` — CI provides none, tests should mock/stub accordingly
