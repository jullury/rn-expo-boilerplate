# Feature Selection & API Provider Setup Design

Date: 2026-05-25
Status: Draft (approved in chat, pending final file review)
Owner: AI pairing session

## 1) Context

This Expo boilerplate already includes production-focused foundations (auth scaffolding, feature flags, observability abstractions, CI/release setup, runbooks). The next requirement is to let users configure the project immediately after clone by running a setup command.

User intent:

- After clone, user runs `pnpm setup`.
- User chooses one API provider from `supabase`, `convex`, `firebase`, or `other`.
- User chooses feature toggles to enable/disable.
- Re-running setup should be safe (reconfigure managed outputs only).

## 2) Goals and Non-Goals

### Goals

- Provide an interactive `pnpm setup` wizard for initial project configuration.
- Support exactly one active API provider at a time.
- Support feature toggles for:
  - auth
  - analytics
  - error-reporting
  - push-notifications
  - payments
- Keep app code provider-agnostic via a stable adapter interface.
- Implement safe reconfigure behavior on repeated setup runs.
- Generate a custom provider adapter stub when `other` is selected.

### Non-Goals

- Runtime provider switching inside a running app.
- Automatic deletion of unused provider implementations/dependencies.
- Full backend integration for every provider in this step.

## 3) Decisions Made

1. Setup command name: `pnpm setup`.
2. “Other” provider behavior: generate a template adapter stub.
3. Provider selection mode: exclusive (one provider only).
4. Feature menu: core set (auth, analytics, error-reporting, push-notifications, payments).
5. Re-run policy: safe reconfigure (update generated/managed files only).

## 4) Architecture

### 4.1 Provider Abstraction

Introduce/standardize a stable provider contract:

- `ApiProviderAdapter` (shared interface).
- Provider-specific implementations:
  - Supabase adapter
  - Convex adapter
  - Firebase adapter
  - Custom adapter (generated stub when needed)
- A resolver/factory returns the active adapter using generated setup configuration.

Application features consume the adapter contract only, not provider SDKs directly.

### 4.2 Setup-Driven Configuration

`pnpm setup` is the source of setup intent and writes canonical setup state into:

- `app.setup.json`

Runtime reads generated TypeScript artifacts derived from this state.

## 5) Setup Wizard Flow

1. Detect existing setup (`app.setup.json` if present).
2. Prompt user for provider:
   - `supabase`
   - `convex`
   - `firebase`
   - `other`
3. Prompt user for feature toggles:
   - auth
   - analytics
   - error-reporting
   - push-notifications
   - payments
4. Validate inputs against setup schema.
5. Write/update managed outputs.
6. Print post-setup guidance:
   - dependency install hints
   - required env vars by provider
   - verification commands

## 6) File Layout and Ownership Boundaries

### 6.1 Generated/Managed Files

- `app.setup.json`
- `src/lib/setup/generated/provider-selection.ts`
- `src/lib/setup/generated/feature-flags.ts`
- `src/lib/api/providers/generated/adapter-resolver.ts`
- `src/lib/api/providers/custom/*` (when `other` selected)

Managed files include a header marker:

`// @generated by pnpm setup - DO NOT EDIT`

### 6.2 Handwritten/Stable Files

- `src/lib/api/providers/types.ts` (adapter contract)
- Provider implementation code under non-generated provider folders
- Feature modules consuming adapter + feature flag exports

Setup must not modify handwritten files outside managed boundaries.

## 7) Safe Reconfigure Policy

On re-run:

1. Read current setup and use current values as prompt defaults.
2. Overwrite managed files only.
3. If an expected managed file is missing header marker or appears manually modified:
   - warn
   - create `*.setup.bak`
   - skip overwrite unless explicitly confirmed for that file
4. If provider changes:
   - update generated resolver to new active provider
   - keep old provider files intact (non-destructive)
   - print cleanup suggestions for potentially unused deps/files

## 8) Validation and Error Handling

### 8.1 Schema Validation

Use zod (or existing validation stack) for:

- provider id enum
- feature key enum
- config shape versioning

### 8.2 Runtime Safety

- If setup config is missing/invalid at runtime, fall back to safe default behavior and emit a warning.
- CLI should fail fast on invalid generated output and provide actionable next steps.

## 9) Testing Strategy

### Unit Tests

- Setup config parse + validation.
- Adapter resolver returns expected provider implementation.
- Feature generation behavior and typing.

### CLI Integration Tests

- First run generates required managed files.
- Re-run updates managed files only.
- `other` provider generates custom adapter stub.
- Marker-missing path triggers backup + skip/confirm flow.

### Project Verification

- `pnpm lint`
- `pnpm typecheck`
- `pnpm test`

## 10) Migration and Developer Experience

### README updates

- Post-clone flow:
  1. `pnpm install`
  2. `pnpm setup`
- Provider-specific env guidance.
- Reconfigure explanation and managed-file rules.

### Backward Compatibility

- Existing projects without `app.setup.json` should continue to run via safe fallback defaults.
- Setup command should bootstrap missing generated artifacts for older clones.

## 11) Risks and Mitigations

- Risk: managed/unmanaged boundary confusion.
  - Mitigation: strict generated markers + docs + backup behavior.
- Risk: provider SDK differences leak into app layer.
  - Mitigation: enforce adapter contract at boundary.
- Risk: setup drift over time.
  - Mitigation: version setup schema and test CLI integration paths.

## 12) Acceptance Criteria

- Running `pnpm setup` allows selecting one provider and core feature toggles.
- `app.setup.json` and generated files are created/updated consistently.
- Selecting `other` creates a custom adapter stub with TODOs.
- Re-running setup safely reconfigures without destructive edits to handwritten files.
- Resolver routes app calls through active adapter.
- Lint, typecheck, and tests pass after implementation.

## 13) Open Implementation Notes

- Keep generated artifacts small and deterministic.
- Prefer explicit imports over dynamic runtime string-eval patterns.
- Keep provider packages optional where feasible; print clear install hints in setup output.
